# SchemaPack Agent 课题 5 后续深化实施文档

> **历史文档**：本文保留当时的需求与执行语境，不作为当前能力清单。当前实施状态见 [`../project_status.md`](../project_status.md)。

## 0. 文档目的

本文档用于指导 Codex 在当前 SchemaPack Agent 项目基础上继续深化课题 5「数据格式标准化转换智能体」。

当前项目已经完成课题 5 主体链路：

```text
UIR -> Schema -> Mapping -> Transform -> Canonical -> Render -> Validate -> Manifest -> ZIP
```

下一阶段不再大幅扩展课题边界，不重做解析、OCR、清洗、归一、完整 RAG 或模型训练，而是围绕以下五个方向补强：

1. 生成验收证据闭环文档。
2. 新增采购类 `procurement_doc` schema/template。
3. 深化真实 UIR 映射与人审知识包闭环。
4. 增加 chunk 分段检索效果评测。
5. 优化前端证据链展示。
6. 可选补充 LLM fallback 评测，但不默认依赖 LLM。

最终目标是让项目从“功能已经完成”进一步变成“有真实数据、有量化评测、有坏例闭环、有前端展示、有验收证据”的完整课题 5 成品。

------

## 1. 总体原则

### 1.1 不要越界扩展生产边界

当前生产系统以 UIR 为输入，不负责原始 PDF、Word、Excel、图片、扫描件、OCR 的生产解析。

允许继续维护 `examples/real_world/` 下的离线真实 UIR 构建工具链，但不要把 raw source parsing 扩展为生产 API。

禁止新增以下生产主功能：

```text
- 生产级 PDF/OCR 解析接口
- 完整 RAG/vector database 系统
- 完整模型训练系统
- 全自动 LLM 映射自动采纳
- 清洗、归一、实体链接等其他课题主体能力
```

### 1.2 保持当前主链路稳定

任何修改不得破坏：

```text
UIR -> Schema -> Mapping -> Transform -> Canonical -> Render -> Validate -> Manifest -> ZIP
```

不得改坏现有 API、现有包结构、现有报告名称。

新增能力应以“可选增强”为主，默认配置保持旧行为兼容。

### 1.3 坚持规则优先、人审闭环

映射策略优先级应保持：

```text
exact -> alias -> regex -> type -> fuzzy -> optional LLM suggestion -> review_required
```

LLM 只能作为建议来源，不得自动接受。

所有低置信、歧义、多金额、多日期、疑似 badcase 的映射结果都必须进入 review_required。

### 1.4 所有成果必须可验证

每个阶段完成后都要提供：

```text
- 代码变更
- 测试用例
- 运行命令
- 生成报告
- 验收标准
```

新增脚本必须输出 JSON 与 Markdown 两种报告，便于机器校验和人工展示。

# Phase 1：新增 procurement_doc schema/template

## 11. 背景

当前真实 UIR 数据集已经包含采购类样本，但采购类样本在 API 评测中仍借用 `general_doc` schema/template。

这会导致两个问题：

1. 采购文档的字段映射不够精准。
2. 真实评测的说服力不足。

因此需要新增专用：

```text
procurement_doc_v1.json
procurement_doc_base_v1.json
```

并纳入 catalog、评测、真实数据执行流程。

## 12. 目标

新增采购类文档的 schema 与基础 mapping template，使采购类真实样本不再借用 `general_doc`。

# Phase 2：真实 UIR 映射与人审知识包闭环

## 20. 背景

真实数据评测中仍存在缺失或歧义字段。这不是错误，而是真实场景下字段映射的正常难点。

下一步要把这些问题显式纳入人审与知识包闭环：

```text
ambiguous/missing mapping
-> review_required
-> approve/reject
-> knowledge candidate
-> draft knowledge pack
-> active knowledge pack
-> rerun evaluation
-> compare before/after
```

## 21. 目标

新增一套“真实映射改进闭环”流程与报告，证明系统能通过人审沉淀映射知识，同时保持 badcase 不被错误激活。

## 22. 建议新增文件

```text
scripts/eval_real_world_knowledge_loop.py
reports/real_world_knowledge_loop_report.json
reports/real_world_knowledge_loop_report.md
docs/real_world_knowledge_loop.md
```

## 23. 脚本职责

`scripts/eval_real_world_knowledge_loop.py` 应完成：

1. 读取真实 UIR 样本。
2. 执行 baseline eval。
3. 收集 mapping report 中的 review_required、missing required、ambiguous 字段。
4. 根据人工配置或 fixture 模拟 approve/reject。
5. 生成 knowledge candidates。
6. 创建 draft pack。
7. 激活 active pack。
8. 重新执行同一批样本。
9. 输出 before/after 指标对比。
10. 检查 badcase violation 是否仍为 0。

## 24. 人审 fixture 设计

新增：

```text
examples/real_world/review_fixtures/procurement_review_decisions.jsonl
```

每行示例：

```json
{
  "doc_id": "real_procurement_001",
  "source_field": "采购项目名称",
  "target_field": "project_name",
  "decision": "approve",
  "reason": "采购类公告中该字段稳定表示项目名称",

```

```json
}
```

## 26. Markdown 报告结构

## 27. 关键约束

### 27.1 不允许自动把所有歧义字段填满

真实歧义必须保留。只有经过 fixture 明确 approve 的候选才能进入 knowledge pack。

### 27.2 Badcase 必须优先于知识包

如果某个新 alias 命中 forbidden pair 或 badcase rule，应阻止激活或在 effective template 中不生效。

### 27.3 旧任务快照不能漂移

在 knowledge pack 激活前已执行的 task，其 snapshot 与历史 package 不应被新知识包修改。

5. badcase alias 不会被 active pack 生效。
6. 激活知识包后旧 task snapshot 不变。

## 30. 背景

## 31. 目标

## 32. 基本思路

不需要调用 LLM。

不需要构建真正 RAG。

## 35. 检索算法

第一版使用确定性检索即可：

1. 对 query 分词或简单字符 n-gram。
2. 对 chunk text、summary、keywords、title_path 计算得分。
3. 命中 expected_terms 加权。
4. source_block_ids 与 expected_block_ids 有交集则认为召回成功。
5. 输出 Recall@k、MRR、nDCG@k。

不要新增重量级依赖。可使用 Python 标准库实现。
