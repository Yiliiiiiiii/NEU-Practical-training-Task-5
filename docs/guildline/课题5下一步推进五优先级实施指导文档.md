## 0.3 绝对边界与禁止事项

### 允许做

- 改进 schema / template / mapping rules。
- 增加真实 UIR 样本。
- 增加 gold labels、badcases、retrieval queries。
- 增加评测脚本和报告生成脚本。
- 增加内容组织策略对比。
- 增加知识包闭环评测。
- 增加成果包下游导出脚本。
- 增加前端展示已有报告和新增报告。
- 增加测试。
- 更新文档。

### 不应做

- 不要把 raw PDF / Word / Excel / image / scan / OCR 解析做成生产 runtime 功能。
- 不要把课题 2 的多源异构解析能力混入课题 5 主服务。
- 不要实现完整向量数据库、完整 RAG 后端、在线检索服务。
- 不要实现模型训练、微调、问答对大规模合成流水线。
- 不要让 LLM fallback 自动接受映射。
- 不要把 API key / LLM key 放进 task options、fixtures、reports、logs、committed `.env`。
- 不要为了提高 strict validation 而简单把所有字段改成 optional。
- 不要为了提高 mapping recall 而让高风险 fuzzy match 自动通过。
- 不要为了通过评测而删除 badcase 或降低 badcase 判断。
- 不要修改旧 task snapshot。
- 不要破坏已有 package spec。

## 1.6 Step 3：general_doc 强化方案

### 1.6.1 general_doc 当前问题推测

`general_doc` 试图覆盖非常广的普通文档，容易出现两个问题：

1. schema 太泛，无法准确表达不同类型普通文档。
2. schema 太硬，把不同文档都压到同一套必填字段，导致 strict validation 失败。

### 1.6.2 推荐策略

采用“稳定主 schema + subtype hint + 温和必填”的方式，不要马上拆成大量 schema。

建议保留 `general_doc` 作为通用入口，但在字段中加入：

```text
document_subtype
```

可选值建议：

```text
service_guide
project_guide
application_flow
notice
announcement
manual
other
```

### 1.6.3 general_doc 建议字段

建议目标字段分为三层。

#### 核心必填字段

```text
title
source_name 或 source_site
main_content 或 body
```

#### 条件字段

```text
issuer
published_at
document_subtype
summary
service_object
application_conditions
application_materials
process_steps
deadline
contact
attachments
```

#### 可选增强字段

```text
keywords
target_audience
related_links
source_url
legal_basis
notes
```

为以下字段补充中文别名：

### 1.6.5 general_doc regex 建议

对正文中常见结构用 regex 提取：

```text
发布机构[:：]\s*(.+)
发布时间[:：]\s*(\d{4}[年/-]\d{1,2}[月/-]\d{1,2}日?)
联系电话[:：]\s*([0-9\-—]{6,})
咨询电话[:：]\s*([0-9\-—]{6,})
```

### 1.6.6 不允许的做法

不要直接把 `title`、`main_content` 这种核心字段改成 optional。

不要把所有无法识别的字段都塞进 `summary` 以假装通过。

不要把明显有歧义的字段高置信映射到错误目标。

## 4.6 安全规则

1. 只有 approved review 可以生成可接受 candidate。
2. candidate 只有被 accept 后才能进入 draft pack。
3. 只有 active pack 影响 future tasks。
4. draft / pending pack 不影响 future tasks。
5. rejected candidate 必须留痕，但不得激活。
6. badcase 命中的 alias 不得进入 active pack。
7. 新 pack 激活不得改变旧 task snapshot。
8. 新 pack 激活后，旧 package 的 metadata / canonical / mapping report 不得变化。

# 11. 风险与处理策略

## 11.1 指标提升不明显

如果增强 alias / regex 后 mapping recall 仍不高：

1. 先看 gap analysis 是否定位准确。
2. 检查 source candidates 是否提取得到目标信息。
3. 如果 candidate extraction 没提取到，应该增强 candidate extraction，而不是只加 alias。
4. 如果 source 文档本身不包含字段，应在 gold label 中标记不可自动映射，不要硬凑。
5. 如果字段存在但表达极度含糊，应保持 review-required。

## 11.2 strict validation 与真实语义冲突

如果某类文档天然没有某字段，不要强行要求 strict pass。应判断 schema 是否设计过强。

调整原则：

```text
真实文档普遍存在的字段，可以 required
只在部分 subtype 存在的字段，应 conditional 或 optional
无法稳定抽取的字段，应 review-required 或 optional with warning
```

## 11.3 数据采集耗时

优先扩展 policy_doc 和 procurement_doc，因为二者公开来源多、字段相对稳定。meeting_doc 次之，general_doc 最后。

## 11.4 前端改动过大

前端只做报告展示和 demo 操作便利化，不要重构整个 UI。

## 11.5 OpenAPI drift

任何 API 改动后都要导出 OpenAPI，并在文档中解释新增路径或 schema 变化。
