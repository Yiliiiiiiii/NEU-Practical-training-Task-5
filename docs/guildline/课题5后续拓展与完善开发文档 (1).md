## 1. 当前项目与课题 5 要求的符合度判断

### 1.1 课题 5 的本质要求

课题 5「数据格式标准化转换智能体」要求实现的是治理流水线中的“转换组织”环节。它不是原始文件解析器，不负责 OCR，不负责清洗归一，不负责最终质量评分，而是接收上游已经治理好的统一中间表示，再输出下游可直接消费的标准成果包。

课题 5 的核心能力可以拆为五类：

1. **Schema 驱动的字段映射**
   - 加载目标 Schema。
   - 加载映射模板。
   - 从 UIR 中抽取源字段候选。
   - 将源字段对齐到目标字段。
   - 支持 exact、alias、regex、type、fuzzy、LLM fallback 等策略。
   - 疑难项给置信度并进入人工复核。

2. **字段转换与结构化输出**
   - 字段重命名。
   - 字段合并。
   - 字段拆分。
   - 日期、数字、枚举、默认值等基础转换。
   - 输出符合目标 Schema 的结构化 JSON。

3. **面向下游的内容组织**
   - 分段 chunks。
   - 内容标签。
   - 管理标签。
   - 质量标签。
   - chunk 摘要。
   - 关键词。
   - 实体标签。
   - 原文回链。

4. **双形态成果包**
   - 机器可读：JSON、chunks.jsonl、reports。
   - 人类可读：Markdown。
   - 包含 manifest、checksum、verifier_report。
   - 包结构完整，可被下游系统一键读取。

5. **可追溯、可评测、可复现**
   - mapping_report。
   - transform_report。
   - validation_report。
   - manifest。
   - verifier_report。
   - production-like evaluator。
   - badcase 回归。

当前项目还没有完全满足“完整课题 5 深化交付”的部分主要是：

### 2.2 明确禁止做的事情

低智能模型执行时必须严格禁止以下行为：

1. **禁止重构主链路**

不要把已经稳定的链路：

```text
UIR -> Schema -> Mapping -> Transform -> Canonical -> Render -> Validate -> Manifest -> ZIP
```

改成另一套架构。

2. **禁止实现原始文件解析**

不要实现：

- PDF 解析。
- Word 解析。
- Excel 解析。
- OCR。
- 图片识别。
- PPT 解析。
- HTML 爬取。

课题 5 的输入就是 UIR，不是原始文件。

3. **禁止越界实现完整课题 11**

可以做“面向课题 5 的轻量内容组织”，但不要实现完整的独立 RAG 分段智能体，不要做复杂 Recall@k/nDCG 检索实验，除非作为可选 smoke test。

4. **禁止让 LLM 自动修改最终结果**

真实 LLM fallback 只能给 suggestion，不能直接自动接受。所有 LLM 建议必须进入人工复核或以 `review_required` 标记输出。

5. **禁止破坏已有 evaluator 指标**

任何新功能不得导致：

- production-like evaluator 跑不通。
- gold 指标下降。
- badcase 失败。
- package verifier 失败。
- 原有 API 不兼容。

6. **禁止删除原有示例数据和报告**

不得删除：

- `examples/demo/`
- `examples/production_like/`
- `reports/production_like_eval_report.*`
- `docs/openapi.json`
- `docs/demo_workflow.md`

## 16.6 具体算法规则

### 16.6.1 token 估算

不要引入 tokenizer 依赖。使用简单估算：

```python
def estimate_tokens(text: str) -> int:
    # 中文按 1.5 字符约等于 1 token，英文按 4 字符约等于 1 token
    if not text:
        return 0
    cjk_count = sum(1 for ch in text if "\u4e00" <= ch <= "\u9fff")
    other_count = len(text) - cjk_count
    return max(1, int(cjk_count / 1.5 + other_count / 4))
```

### 16.6.2 摘要生成

默认不用 LLM。规则如下：

1. 去掉多余空白。
2. 按中文句号、英文句号、分号、换行切分。
3. 如果第一句长度在 20~120 字之间，使用第一句作为摘要。
4. 如果第一句太短，则合并前两句。
5. 如果仍不合适，则截取前 120 字并加 `...`。
6. 摘要不得生成原文没有的信息。

伪代码：

```python
def summarize_text(text: str, max_chars: int = 120) -> str:
    cleaned = normalize_spaces(text)
    if not cleaned:
        return ""
    sentences = split_sentences(cleaned)
    if sentences:
        summary = sentences[0]
        if len(summary) < 20 and len(sentences) > 1:
            summary = summary + sentences[1]
        if len(summary) <= max_chars:
            return summary
    return cleaned[:max_chars].rstrip() + ("..." if len(cleaned) > max_chars else "")
```

### 16.6.3 关键词提取

默认不用 jieba，避免新依赖。规则如下：

1. 提取中文连续 2~8 字、英文数字连续词。
2. 去除停用词。
3. 统计词频。
4. 优先保留 schema 字段名、标题词、metadata 中出现的词。
5. 最多输出 8 个关键词。

停用词示例：

```python
STOPWORDS = {
    "的", "了", "和", "与", "及", "或", "在", "对", "为", "是", "本", "该", "等",
    "this", "that", "and", "or", "the", "a", "an", "of", "to", "in", "for"
}
```

低智能模型实现时可以先做简单 regex：

```python
re.findall(r"[A-Za-z][A-Za-z0-9_\-]{2,}|[\u4e00-\u9fff]{2,8}", text)
```

```python
    "general_doc": {
        "overview": ["概述", "简介", "背景"],
        "requirement": ["要求", "标准", "指标"],
        "result": ["结果", "结论", "产出"]
    }
}
```

输出规则：

- 命中关键词则添加对应 tag。
- schema_id 归属也可以加一个大类 tag。
- 如果没有命中，添加 `general`。
- tags 去重并排序，保证可复现。

### 16.6.5 管理标签 management tags

管理标签按规则生成，不需要语义判断。

建议包括：

```text
schema:<schema_id>
template:<template_id>
doc:<doc_id>
task:<task_id>
chunk_index:<index>
```

如果 schema/version 可用，则增加：

```text
schema_version:<version>
template_version:<version>
```

### 16.6.6 质量标签 quality tags

质量标签根据现有 report 和 chunk 状态生成。

建议规则：

- 有 source_block_ids：`source_linked`
- source_links 非空：`anchor_linked`
- validation_report 无 error：`validation_passed`
- validation_report 有 error：`validation_has_errors`
- mapping_report 中有 review_required：`mapping_review_required`
- chunk text 为空：`empty_text`
- chunk token_estimate 太长：`overlong_chunk`
- chunk token_estimate 太短：`short_chunk`
- chunk 有 summary：`summarized`
- chunk 有 keywords：`keyworded`

### 16.6.7 实体标签 entity_tags

不要实现完整实体链接。只做轻量占位，体现课题 5 “可写入标准实体标签”的接口能力。

实体标签来源优先级：

1. canonical structured data 中字段名包含：
   - `entity`
   - `company`
   - `organization`
   - `person`
   - `party`
   - `department`
   - `机构`
   - `部门`
   - `人员`
   - `甲方`
   - `乙方`

2. metadata 中类似字段。

3. mapping_report 中目标字段名带实体语义。

输出示例：

```json
{
  "text": "国家电网有限公司",
  "entity_type": "organization",
  "normalized_id": null,
  "source": "schema_field",
  "confidence": 0.8
}
```

注意：

- `normalized_id` 可以为空。
- 不要硬编实体标准库。
- 不要声称已经完成课题 10。

### 16.6.8 source_links

如果 UIR block 有 anchor 信息，则转换为 source_links。

如果没有 anchor，也至少保留：

```json
{
  "block_id": "block_001",
  "source_path": "blocks[0]",
  "page_no": null,
  "bbox": null,
  "anchor_text": "文本前 20 字"
}
```

目标是保证每个 chunk 可以回链到原始 UIR block。

## 16.7 Pipeline 接入点

当前主链路大概率如下：

```text

  -> load schema/template
  -> extract candidates
  -> map
  -> transform
  -> build canonical
  -> render
  -> validate
  -> package
```

Phase 16 后应改为：

```text
TaskExecutionService.execute_task
  -> load schema/template
  -> extract candidates
  -> map
  -> transform
  -> build canonical
  -> render basic outputs
  -> validate basic outputs
  -> organize chunks
  -> validate organized outputs
  -> package enriched outputs
```

也可以先做：

```text
render -> organize -> validate -> package
```

两种都可以，但最终 validation 要看到 enriched chunks。

### 16.7.1 推荐实现方式

1. RenderService 仍然负责生成基础 chunks。
2. ChunkOrganizerService 接收基础 chunks，返回 enriched chunks。

4. PackageVerifierService 检查 enriched chunks。
5. 原有 RenderService 不承担 tag/summary 逻辑，避免职责膨胀。

# Phase 19：真实 LLM Fallback Adapter（可选）

## 19.1 阶段目标

在保持默认关闭的前提下，为 mapping 疑难项提供真实 LLM suggestion 能力。

注意：LLM 不是主流程，不能成为必需依赖。

目标架构：

```text
LLMFallbackAdapter interface
  - DisabledLLMAdapter
  - StubLLMAdapter
  - OpenAICompatibleLLMAdapter
```

## 19.2 阶段边界

### 19.2.1 允许

- 支持 OpenAI-compatible API。
- 通过环境变量配置 base_url/api_key/model。
- 记录 prompt_hash、model、latency、cost estimate。
- 输出 suggestion。

### 19.2.2 禁止

- 禁止默认开启。
- 禁止无审核自动接受。
- 禁止把 API key 写入代码或日志。
- 禁止联网失败导致整个任务失败。
- 禁止突破 badcase filtering。

## 19.3 配置设计

`.env.example`：

```text
LLM_FALLBACK_ENABLED=false
LLM_FALLBACK_PROVIDER=openai_compatible
LLM_FALLBACK_BASE_URL=
LLM_FALLBACK_API_KEY=
LLM_FALLBACK_MODEL=
LLM_FALLBACK_TIMEOUT_SECONDS=20
LLM_FALLBACK_MAX_RETRIES=1
```

`config.py` 中增加对应 settings。

## 19.4 Adapter 接口

```python
class LLMMappingSuggestion(BaseModel):
    target_field: str | None
    confidence: float
    reason: str
    raw_response_hash: str | None = None
    review_required: bool = True


class LLMFallbackAdapter(Protocol):
    def suggest_mapping(
        self,
        *,
        source_field_name: str,
        value_sample: Any,
        target_fields: list[TargetField],
        schema_id: str,
        context: dict[str, Any],
    ) -> LLMMappingSuggestion:
        ...
```

输出必须强制：

```python
review_required = True
```

## 19.5 Prompt 设计

系统 prompt 必须约束：

```text
你是数据治理字段映射助手。你只能从给定 target_fields 中选择一个字段，不能创造新字段。
如果不确定，返回 null。你必须给出 confidence。你的建议不会自动生效，只会进入人工审核。
```

用户 prompt 输入：

```json
{
  "source_field_name": "签署日期",
  "value_sample": "2026年6月11日",
  "target_fields": [
    {"name":"contract_date", "type":"date", "description":"合同签署日期"},
    {"name":"amount", "type":"number", "description":"合同金额"}
  ]
}
```

要求模型输出 JSON：

```json
{
  "target_field": "contract_date",
  "confidence": 0.86,
  "reason": "source field 表示签署日期，与 contract_date 语义和类型一致"
}
```

## 19.6 MappingService 接入规则

Mapping strategy order 仍然是：

```text
exact -> alias -> regex -> type -> fuzzy -> llm_suggestion -> review/failed
```

LLM 只在以下条件触发：

- `LLM_FALLBACK_ENABLED=true`
- deterministic 策略无法高置信匹配
- source field 不在 badcase deny list
- target schema fields 可用

LLM 输出不进入 accepted mapping，只进入 review item：

```json
{
  "strategy": "llm_suggestion",
  "target_field": "...",
  "confidence": 0.82,
  "review_required": true
}
```

## 19.7 测试要求

必须覆盖：

1. 默认 disabled，不调用 LLM。
2. stub adapter 输出 review_required。
3. OpenAI-compatible adapter 缺 API key 时不崩溃，返回 disabled/error suggestion。
4. LLM suggestion 不自动 accepted。
5. badcase 命中时不调用 LLM。
6. prompt hash 被记录。

## 20.2 smoke_rag_ingest.py

### 20.2.1 功能

输入一个 package zip 或 package directory，执行：

1. 校验 manifest。
2. 读取 chunks.jsonl。
3. 构建一个简单内存索引。
4. 用固定 query 做 keyword match。
5. 返回命中的 chunk。
6. 检查命中 chunk 有 source link。

### 20.2.2 命令示例

```powershell

```

输出示例：

```text
package_valid: true
chunks_loaded: 12
query: 合同金额
top_hit_chunk_id: task_x_chunk_0003
source_linked: true
passed: true
```

### 20.2.3 不要做

- 不要引入向量数据库。
- 不要调用 embedding API。
- 不要做复杂 RAG。

### 20.3.1 功能

从成果包导出训练语料格式，例如 jsonl：

```json
  "text": "...",
  "metadata": {
    "schema_id": "contract_doc",
    "tags": ["amount", "source_linked"],
    "source_block_ids": ["block_1"]
  }
}
```

### 20.3.2 命令示例

```powershell
python scripts\export_training_corpus.py --package reports\packages\xxx.zip --out reports\training_corpus.jsonl
```

```text
SchemaPack Agent 是面向课题 5「数据格式标准化转换智能体」的可运行服务。系统以上游统一中间表示 UIR 为输入，通过 SchemaService、CandidateService、MappingService、TransformService、CanonicalService、RenderService、ChunkOrganizerService、ValidationService 与 PackageService 组成受控流水线，将异构治理结果转换为结构化 JSON、人读 Markdown、面向下游的 chunks.jsonl 以及包含 manifest/checksum/verifier 的标准成果包。
```

# 9. 不足与后续研究方向

即使完成上述 Phase，仍可以诚实说明以下不足：

1. 输入依赖上游 UIR，不处理原始 PDF/Word/OCR。
2. 内容组织是课题 5 内部轻量实现，不等同于完整课题 11 智能分段智能体。
3. 实体标签只提供占位与接口，不等同于完整课题 10 实体链接。
4. 真实 LLM fallback 默认关闭，主要用于疑难映射建议。
5. production-like dataset 是合成数据，真实企业数据仍需扩充评测。
6. 前端偏 demo workbench，不是完整生产后台。
7. 权限、租户、审计日志可以进一步生产化。

这些不足不影响课题 5 主体成立，反而能体现边界清晰。
