# 课题 5 SchemaPack Agent 四项深化实施指导文档

> 目标读者：Codex / 代码执行智能体  
> 适用项目：SchemaPack Agent（课题 5：数据格式标准化转换智能体）  
> 本文目标：在不重构主架构、不越界实现课题 2 原始文件解析的前提下，把当前项目从“功能跑通”推进到“能力可证明、可评测、可展示”。

---

## 0. 当前项目判断与本次任务边界

当前项目已经具备课题 5 的核心服务链路：

```text
UIR -> Schema -> Mapping -> Transform -> Canonical -> Render -> Validate -> Manifest -> ZIP
```

本次不要推翻现有架构，也不要把重点转向 PDF / Word / OCR / 图片解析。项目生产边界仍然是 **UIR 输入**。真实网页、PDF、DOCX 等来源只用于离线构造真实 UIR 评测集，不扩展生产 API 边界。

本次只做四个深化方向：

1. 把真实 UIR 数据集从“能跑通”推进到“能证明能力”。
2. 新增 `procurement_doc` 专用 Schema / Template。
3. 加强内容组织评测，而不是只展示 chunks。
4. 完善人审与知识包闭环。

最终产出不是单纯功能堆叠，而是形成一套可复现的评测证据：字段映射指标、真实 badcase、领域 Schema 扩展收益、chunk 策略下游检索收益、人工审核知识沉淀前后对比。

# 第一部分：把真实 UIR 数据集从“能跑通”推进到“能证明能力”

## 3. 目标

当前真实 UIR 数据集已经可以证明系统能导入、执行、生成包、通过 verifier。但这还不够。下一步要证明：

```text
- 字段映射是否准确；
- 必填字段覆盖率如何；
- 哪些字段因为真实文档歧义而进入 review_required；
- 哪些失败属于真实 badcase，而不是系统崩溃；
- 人工审核 / 知识包激活后是否能改善后续映射。
```

最终要让真实 UIR 数据集从“跑通样例”变成“课题 5 能力评测集”。

---

## 4. 真实 UIR 映射金标设计

### 4.1 新增金标文件

新增：

```text
examples/real_world/gold/mapping_gold.jsonl
```

每行一个 JSON 对象。建议结构：

```json
{
  "doc_id": "real_policy_001",
  "doc_type": "policy_doc",
  "schema_id": "policy_doc",
  "template_id": "policy_doc_base_v1",
  "expected_mappings": [
    {
      "source_name": "发文机关",
      "source_path": "metadata.issuer",
      "target_field": "issuer",
      "required": true,
      "match_type": "exact_or_alias",
      "gold_status": "accepted"
    }
  ],
  "expected_review_required": [
    {
      "source_name": "金额",
      "target_field_candidates": ["budget_amount", "award_amount"],
      "reason": "multiple_amounts_in_procurement_notice"
    }
  ],
  "known_badcases": [
    {
      "case_id": "ambiguous_date_001",
      "source_name": "截止时间",
      "forbidden_target_field": "announcement_date",
      "reason": "deadline_date_should_not_be_mapped_to_announcement_date"
    }
  ]
}
```

字段说明：

| 字段 | 含义 |
| --- | --- |
| `doc_id` | 与真实 UIR 文件中的 `doc_id` 一致 |
| `doc_type` | 文档类型，例如 `policy_doc`、`procurement_doc`、`meeting_doc`、`general_doc` |
| `schema_id` | 应使用的目标 schema |
| `template_id` | 应使用的 base template |
| `expected_mappings` | 人工确认的源字段到目标字段金标 |
| `expected_review_required` | 应进入人工复核的歧义项 |
| `known_badcases` | 禁止自动接受的错误映射 |

### 4.2 金标规模要求

最低要求：

```text
- 覆盖所有 16 个真实 UIR 样本；
- 每个样本至少标注 3 个字段；
- 每类文档至少有 1 个 known_badcase；
- procurement_doc 至少覆盖金额、日期、采购人、代理机构、供应商等高价值字段；
- 所有 gold 文件必须通过 JSONL 解析测试。
```

推荐要求：

```text
- 每个样本标注 5–10 个字段；
- 每个 expected_mapping 都带 source_path 或 source_name；
- 每个 review_required 项都有 reason；
- badcase 尽量来自真实失败，而不是凭空构造。
```

## 5. 真实 UIR 映射评测脚本

### 5.1 新增脚本

新增：

```text
scripts/eval_real_world_mapping.py
```

脚本目标：读取真实 UIR 金标，调用当前真实 API 或复用服务层，统计字段映射能力。

建议支持参数：

```powershell
python scripts/eval_real_world_mapping.py `
  --base-url http://127.0.0.1:8000 `
  --gold examples/real_world/gold/mapping_gold.jsonl `
  --out-json reports/real_world_mapping_eval_report.json `
  --out-md reports/real_world_mapping_eval_report.md
```

如果项目已有 API key 逻辑，支持：

```powershell
--api-key your-dev-key
```

如果后端未启动，脚本可以提供 `--offline` 模式直接调用服务层，但优先走 HTTP API，因为真实 UIR 评测本身就是要证明端到端可用。

### 5.2 脚本流程

伪流程：

```text
1. 加载 mapping_gold.jsonl。
2. 对每个 doc_id 找到对应真实 UIR 文件。
3. 选择 schema_id / template_id。
4. POST /api/v1/documents/import。
5. POST /api/v1/tasks。
6. POST /api/v1/tasks/{task_id}/execute。
7. GET /api/v1/tasks/{task_id}/reports/mapping。
8. GET /api/v1/tasks/{task_id}/reports/validation。
9. GET /api/v1/tasks/{task_id}/package。
10. 与 gold 比对，统计指标。
11. 输出 JSON + Markdown 报告。
```

### 5.3 指标定义

必须输出以下指标：

| 指标 | 说明 |
| --- | --- |
| `document_count` | 参与评测的真实 UIR 数量 |
| `gold_mapping_count` | 人工金标映射总数 |
| `auto_accepted_correct` | 自动接受且命中金标的数量 |
| `auto_accepted_wrong` | 自动接受但不符合金标的数量 |
| `review_required_correct` | 正确进入人工复核的数量 |
| `missing_gold_mappings` | 金标字段未被映射也未进入复核的数量 |
| `mapping_precision` | 自动接受映射中的正确比例 |
| `mapping_recall` | 金标映射被自动接受或合理复核覆盖的比例 |
| `required_field_coverage` | 必填字段覆盖率 |
| `review_required_rate` | 映射项中需要复核的比例 |
| `badcase_violation_count` | 违反 forbidden mapping 的数量 |
| `package_pass_rate` | 成果包 verifier 通过率 |

推荐补充：

```text
- per_doc_type 指标：policy / procurement / meeting / general 分开统计；
- per_field 指标：统计哪些 target_field 最容易失败；
- warning_summary：汇总 validation/mapping warnings；
- unresolved_required_fields：列出缺失必填字段。
```

### 5.4 报告内容

Markdown 报告至少包含：

```markdown
# Real-world Mapping Evaluation Report

## Summary

| Metric | Value |
| --- | ---: |
| document_count | ... |
| mapping_precision | ... |
| mapping_recall | ... |
| required_field_coverage | ... |
| badcase_violation_count | ... |

## Per Document Type

## Per Document Results

## Missing / Ambiguous Fields

## Badcase Violations

## Review Required Evidence

## Package Verification Summary

## Next Recommendations
```

### 5.5 判定标准

初始阶段不要追求所有真实样本 100% 自动映射。合理目标是：

```text
- 所有真实 UIR 能导入和执行；
- 所有 package 能生成并通过 verifier；
- badcase_violation_count = 0；
- 自动接受错误率尽量低；
- 歧义项必须进入 review_required，而不是被错误高置信接受；
- 报告能清楚解释 missing / ambiguous 字段。
```

## 6. 真实 badcase 汇总

### 6.1 新增 badcase 明细

可以新增：

```text
examples/real_world/gold/real_world_badcases.jsonl
```

结构建议：

```json
{
  "case_id": "rw_procurement_amount_001",
  "doc_id": "real_procurement_001",
  "badcase_type": "multiple_amounts",
  "source_evidence": "预算金额、成交金额同时出现",
  "forbidden_auto_mapping": {
    "source_name": "预算金额",
    "target_field": "award_amount"
  },
  "expected_behavior": "review_required_or_map_to_budget_amount",
  "severity": "high"
}
```

### 6.2 badcase 类型建议

至少覆盖：

```text
- ambiguous_date：公告日期 / 截止日期 / 开标日期混淆；
- multiple_amounts：预算金额 / 成交金额 / 最高限价混淆；
- similar_org_roles：采购人 / 代理机构 / 中标供应商混淆；
- irregular_heading：标题层级不规范导致字段候选弱；
- flattened_complex_table：复杂表格被扁平化后 source_path 不稳定；
- general_schema_overload：采购文档套用 general_doc 导致字段缺失。
```

这些 badcase 不一定都要修成自动 accepted。更重要的是：不能错映射，必须保留 review evidence。

# 第二部分：新增 procurement_doc 专用 Schema / Template

## 7. 目标

当前真实采购类样本如果暂时使用 `general_doc` schema/template，会导致采购领域关键字段缺失或含义不清。本阶段要新增专用 `procurement_doc` schema/template，让采购公告 / 中标公告 / 采购结果类真实 UIR 有专门结构。

核心目标：

```text
- 新增 procurement_doc_v1 schema；
- 新增 procurement_doc_base_v1 template；
- 真实采购样本评测时使用 procurement_doc，不再套 general_doc；
- 对比 general_doc 与 procurement_doc 的字段覆盖率提升；
- 保持 catalog governance、version、activation、archival 规则兼容。
```


## 12. 查询金标设计

### 12.1 新增文件

```text
examples/real_world/gold/retrieval_queries.jsonl
```

每行示例：

```json
{
  "query_id": "rw_procurement_001_q1",
  "doc_id": "real_procurement_001",
  "query": "该项目的预算金额是多少？",
```

字段说明：

| 字段 | 含义 |

| `relevant_source_block_ids` | 相关原文块 ID，优先用于自动评测 |
| `relevant_title_path_contains` | 相关标题路径关键词 |
| `relevant_keywords` | 相关内容关键词 |
| `doc_type` | 文档类型 |

### 12.2 数量要求

最低要求：

```text
- 每个真实 UIR 至少 2 个 query；
- 16 个样本至少 32 个 query；
- 每类文档至少 6 个 query；
- procurement_doc 至少覆盖：项目名称、预算金额、采购人、中标供应商、截止日期；
- query 必须可由原文 chunk 回答，不能问原文没有的信息。
```

推荐要求：

```text
- 每个真实 UIR 3–5 个 query；
- 至少一部分 query 需要表格 chunk 才能回答；
- 至少一部分 query 需要标题层级上下文；
- parent_child 策略应该有多粒度评测样本。
```

## 13. 内容组织检索评测脚本

### 13.1 新增脚本

```text
scripts/eval_content_organization_retrieval.py
```

命令示例：

```powershell
python scripts/eval_content_organization_retrieval.py `
  --base-url http://127.0.0.1:8000 `
  --queries examples/real_world/gold/retrieval_queries.jsonl `
  --strategies fixed_window heading_aware source_block_aware table_protect parent_child `
  --out-json reports/content_organization_retrieval_eval.json `
  --out-md reports/content_organization_retrieval_eval.md
```

### 13.2 任务执行参数

每种策略创建任务时传入：

```json
{
  "options": {
    "content_organization": {
      "chunk_strategy": "heading_aware",
      "target_tokens": 768,
      "min_tokens": 128,
      "max_tokens": 1024,
      "overlap_tokens": 80,
      "protect_tables": true,
      "protect_lists": true,
      "protect_code_blocks": true,
      "enable_parent_child": false,
      "enable_light_semantic_boundary": true,
      "summary_mode": "deterministic",
      "keyword_mode": "deterministic"
    }
  }
}
```

不同策略调整：

| 策略 | 参数建议 |
| --- | --- |
| `fixed_window` | `protect_tables=false`、`enable_parent_child=false` |
| `heading_aware` | 标题感知，表格保护开启 |
| `source_block_aware` | 以 UIR source block 为主要边界 |
| `table_protect` | 强制表格整体保护 |
| `parent_child` | `enable_parent_child=true`，评测 child、parent、all 三种 granularity |

具体字段名以现有 `ContentOrganizationOptions` 为准。

### 13.3 检索实现

优先使用零外部依赖或轻依赖方式：

```text
方案 A：纯 Python keyword / token overlap scorer；
方案 B：rank_bm25，如果项目已允许新增依赖；
方案 C：sklearn TfidfVectorizer，如果依赖已存在；
方案 D：本地 embedding，只有在项目已有依赖时再做可选项。
```

推荐先实现方案 A，保证无外部模型、可私有化、可复现。

Scoring 示例：

```text
score = query_keyword_overlap
      + title_path_match_bonus
      + chunk_keywords_match_bonus
      + source_block_id_match_bonus_for_eval_only
```

注意：评测时不能把 gold block id 用于排序，否则作弊。`source_block_id_match_bonus_for_eval_only` 只能用于判断命中，不用于检索排序。

### 13.4 指标定义

必须输出：

| 指标 | 说明 |
| --- | --- |
| `Recall@1` | Top 1 是否包含相关 chunk |
| `Recall@3` | Top 3 是否包含相关 chunk |
| `Recall@5` | Top 5 是否包含相关 chunk |
| `MRR` | 第一个相关 chunk 的倒数排名 |
| `nDCG@5` | 前 5 个结果的排序质量 |
| `answerable_rate` | Top k 中是否存在可回答 chunk |
| `avg_chunk_count` | 平均 chunk 数量 |
| `avg_chunk_tokens` | 平均 token estimate |
| `table_split_violation_count` | 表格被不合理拆断次数 |
| `source_link_coverage` | chunks 是否都有原文回链 |

### 13.5 报告要求

Markdown 报告至少包含：

```markdown
# Content Organization Retrieval Evaluation

## Summary

| Strategy | Recall@3 | Recall@5 | MRR | nDCG@5 | Table Split Violations | Source Link Coverage |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |

## Strategy Comparison

## Per Document Type

## Per Query Failure Cases

## Chunk Quality Statistics

## Recommendation
```

### 13.6 判定标准

初始目标：

```text
- 所有策略都能跑通并生成 chunks；
- heading_aware / table_protect / parent_child 至少有一项指标优于 fixed_window；
- table_protect 的 table_split_violation_count 应低于 fixed_window；
- source_link_coverage 应接近 100%；
- parent_child 的 child 或 all granularity 应在 Recall@5 上有优势或提供解释。
```

不要硬造指标。如果真实结果没有提升，要在报告中如实记录，并指出 query 金标、检索器或分段策略需要继续优化。

# 第四部分：完善人审与知识包闭环

## 14. 目标

当前项目已有 review、knowledge candidates、draft/active knowledge packs、effective template 等能力。下一步要用真实 UIR 证明闭环有效：

```text
真实映射歧义
→ 进入 review_required
→ 人工 approve/reject
→ 生成 alias / knowledge candidate
→ 创建 draft pack
→ 激活 active pack
→ 新任务使用 effective template 命中更好映射
→ 历史任务 snapshot 不变
→ badcase 仍被保护，不会因知识包被错误自动接受
```

这是课题 5 很重要的“治理智能体”特征：不是一次性转换脚本，而是可审计、可沉淀、可回归的转换组织系统。

