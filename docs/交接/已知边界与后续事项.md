# 已知边界与后续事项

## 明确边界

以下能力不属于当前生产 runtime：

- raw PDF/Word/Excel/image upload API；
- OCR 与扫描件识别；
- 完整 RAG/vector database；
- 模型训练、fine-tuning；
- LLM 自动接受 mapping；
- LLM 自动激活 schema/template；
- Webhook；
- SSO、tenant-aware RBAC；
- TLS termination 与 hosted secret management；
- 企业级 model/provider monitoring。

Optional Docling/Unstructured 仅是离线工具。无文本层的扫描 PDF 应返回
`unsupported_scanned_pdf`，不能静默启用 OCR。

## 语义边界

- Package verifier 通过证明结构、manifest、hash、文件存在和可解析。
- 它不证明每个 target field 都通过 strict semantic validation。
- Lineage 证明来源、证据和决策历史可追溯。
- 它不等同于字段语义严格正确。
- Review-required 与 badcase-blocked 状态必须保持可见，不能为了提高通过率
  自动改成 accepted。

## 当前技术债与限制

1. 非采购数据 strict pass 为 17/35，仍有可解释的人工 Review 空间。
2. Lineage graph/summary 当前是 task reports，不进入 Package 1.1 ZIP。
3. Webhook 未实现；CLI、SDK 和 Adapter scaffold 已完成。
4. 当前鉴权适合本地/受控环境，不是完整多租户生产安全模型。
5. Evaluation reports 是特定数据集与时点的证据，需要通过脚本重新生成。

## 建议后续优先级

### P1：继续提升严格语义质量

- 扩充非采购 gold labels 与 badcases。
- 优先处理高频 review-required 字段。
- 保持 required field 与安全门，不通过删字段提高 strict pass。

### P2：Lineage package contract

- 先定义不引用自身 hash 的 package-lineage contract。
- 明确 lineage artifact 的 required/optional 语义。
- 同步 ManifestService、PackageVerifierService 与 downstream contracts。

### P3：生产治理

- SSO/TLS、托管密钥、tenant-aware authorization。
- operator review audit 与告警。
- provider/model evaluation 与运行监控。

### P4：集成扩展

- 如确有业务需求，再实现 Webhook。
- 可发布由 OpenAPI 生成的强类型客户端。
- 继续扩充 adapter 插件与 versioned consumer contracts。

## 禁止性原则

- 不关闭 badcase filter。
- 不让 LLM suggestion 自动 accepted。
- 不让 draft knowledge pack 影响当前 task。
- 不修改历史 task snapshot。
- 不把 lineage 当作语义正确性的替代证明。
- 不把离线上游包装成生产 OCR API。
