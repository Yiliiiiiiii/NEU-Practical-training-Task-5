# 已知边界与后续事项

## 明确边界

以下能力不属于当前生产 runtime：

- raw PDF/Word/Excel/image upload API；
- OCR 与扫描件识别；
- 完整 RAG/vector database；
- 模型训练、fine-tuning；
- LLM 自动接受 mapping；
- LLM 自动激活 schema/template；
- LLM 自动创建或执行 task；
- Webhook；
- SSO、tenant-aware RBAC；
- TLS termination 与 hosted secret management；
- 企业级 model/provider monitoring。

Optional Docling/Unstructured 仅是离线工具。无文本层的扫描 PDF 应返回
`unsupported_scanned_pdf`，不能静默启用 OCR。

## 语义边界

- Package verifier 通过证明结构、manifest、hash、文件存在和可解析。
- 它不证明每个 target field 都通过 strict semantic validation。
- Lineage 证明来源、证据和决策历史可追溯。
- 它不等同于字段语义严格正确。
- Review-required 与 badcase-blocked 状态必须保持可见，不能为了提高通过率自动改成 accepted。
- DeepSeek/LLM suggestion 必须保持 report-only 或 review-required，不能绕过 badcase 和人工治理。

## 当前技术债与限制

1. 当前 non-procurement average recall 为 `0.8063730159`，已超过 `0.78`，但仍未达到 `0.85`。
2. Policy_doc 是当前主要瓶颈：recall `0.6655753968`，strict pass 11/20，issuer/publish_date 与长尾政策字段仍需改进。
3. 当前 review-required 为 16，已低于目标 ≤18；后续仍需避免为追 recall 过度降低 review 门槛。
4. 当前没有独立 production shadow/blind UIR gold corpus，因此不能宣称 blind recall 0.85。
5. Lineage graph/summary 当前是 task reports，不进入 Package 1.1 ZIP。
6. Webhook 未实现；CLI、SDK 和 Adapter scaffold 已完成。
7. 当前鉴权适合本地/受控环境，不是完整多租户生产安全模型。
8. Evaluation reports 是特定数据集与时点的证据，需要通过脚本重新生成。

## 建议后续优先级

### P1：继续提升严格语义质量

- 优先处理 policy_doc 的 issuer、publish_date、effective_date 与长尾政策字段。
- 针对 `real_policy_005` 等 required-missing 样本补充 source-backed safe rule。
- 保持 required field 与安全门，不通过删字段提高 strict pass。

### P2：建立独立 blind/shadow corpus

- 构建与当前 60-sample real-world corpus 去重的 production shadow/blind UIR gold set。
- 明确 blind-set sampling、gold labeling、badcase registry 与冻结规则。
- 只有在独立 blind/shadow report 产生后，才能讨论 0.85 claim。

### P3：Lineage package contract

- 先定义不引用自身 hash 的 package-lineage contract。
- 明确 lineage artifact 的 required/optional 语义。
- 同步 ManifestService、PackageVerifierService 与 downstream contracts。

### P4：生产治理

- SSO/TLS、托管密钥、tenant-aware authorization。
- operator review audit 与告警。
- provider/model evaluation 与运行监控。

### P5：集成扩展

- 如确有业务需求，再实现 Webhook。
- 可发布由 OpenAPI 生成的强类型客户端。
- 继续扩充 adapter 插件与 versioned consumer contracts。

## Topic 5 Phase 3 Boundary

Conversion output assertions are deterministic SchemaPack-scoped checks over Topic 5 converted output. They complement target-schema validation but do not implement Topic 6 quality scoring, grading, semantic fidelity evaluation, or routing recommendations.

Phase 3 explicitly has no quality score, no quality grade, no publication route, no semantic fidelity judgment, no LLM-as-Judge, and no Topic 11 retrieval optimization. The following phase owns cross-artifact completeness, JSON/Markdown/chunk consistency, one-click downstream export, and consumer contract expansion.

## 禁止性原则

- 不关闭 badcase filter。
- 不让 LLM suggestion 自动 accepted。
- 不让 draft knowledge pack 影响当前 task。
- 不修改历史 task snapshot。
- 不把 lineage 当作语义正确性的替代证明。
- 不把离线上游包装成生产 OCR API。
- 不把 `not_run` 的 blind/shadow 报告表述为已达 0.85。
